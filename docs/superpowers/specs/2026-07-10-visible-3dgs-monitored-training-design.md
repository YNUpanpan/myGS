# Visible 3DGS 监控训练与质量早停设计

## 目标

- 只训练 visible 场景，不启动 thermal 训练。
- 使用官方 `graphdeco-inria/gaussian-splatting` 基线，在单张 RTX 5090 上训练。
- 最大训练迭代数为 15000。
- 从 5000 步开始，每 1000 步记录定量质量和固定视角渲染效果。
- 当质量相对上一个检查点明确下降时，在当前检查点停止训练。
- 训练进度、评估结果、停止原因和最佳检查点均可实时读取并可复现。

## 已确认约束

- 服务器：`pch-5090`。
- 项目根目录：`/home/pch/myGS`。
- visible 输入：`/home/pch/myGS/datasets/uav_3dgs/processed/visible`。
- 原始数据保持只读，不移动、不删除、不改名。
- 使用 GPU 0；GPU 1 不参与本次训练。
- 使用项目 Conda 环境 `mygs-3dgs-cu128` 和 PyTorch cu128。
- 3DGS 源码固定为已校验的提交 `54c035f`。
- 启用官方 `--eval` 划分：按已排序图像名每 8 张取 1 张测试图。
- 对 339 张 visible 图像，固定得到 296 张训练图和 43 张测试图。
- 不使用批量删除或递归删除命令，不清理已生成的训练结果。

## 方案选择

采用单 GPU、单训练进程内同步评估与精确早停。

训练在 GPU 0 上连续运行。到达评估检查点时，训练循环暂停，使用同一张 GPU 对固定的 43 张测试图进行评估。评估结束并完成早停判断后，训练继续或在当前检查点退出。该方案保持优化器轨迹连续，也避免异步双 GPU 评估产生的停止延迟。

不采用以下方案：

- 双 GPU 异步评估：可以并行，但质量下降被发现时训练可能已经越过对应检查点。
- 每 1000 步退出并从 checkpoint 重启：会增加启动开销，并改变官方训练循环边界行为。
- 仅解析外部日志并强制终止：完整指标和精确停止点难以保证。

## 训练与评估流程

1. 启动前验证 visible 的 `images/` 和 `sparse/0/` 输入完整。
2. 验证项目 Conda、CUDA、3DGS CUDA 扩展和 GPU 0 可用。
3. 验证 3DGS checkout HEAD 为 `54c035f`，并以幂等方式应用受 Git 跟踪的最小监控补丁。
4. 在正式训练前加载并缓存 LPIPS VGG 所需权重；若无法完成则不启动正式训练。
5. 使用 `--eval` 启动 visible 训练，最大迭代数为 15000。
6. 每 10 步刷新实时状态，包括迭代、百分比、EMA loss、已用时间和预计剩余时间。
7. 在 5000、6000、7000、8000、9000、10000、11000、12000、13000、14000、15000 步同步执行评估。
8. 每个评估点保存 point cloud、训练 checkpoint、全部测试图指标和固定测试视角渲染图。
9. 5000 步只建立质量基准；从 6000 步开始与上一个检查点比较并执行早停判据。
10. 触发早停时保留当前及此前的所有输出，记录停止原因，并标出最佳检查点。
11. 未触发早停时在 15000 步正常结束，并从全部评估点中标出最佳检查点。

## 质量指标和早停判据

每个评估点对全部 43 张固定测试图计算：

- PSNR：越高越好。
- SSIM：越高越好。
- LPIPS（VGG）：越低越好。

相对上一个检查点，同时满足以下条件时判定质量下降：

1. PSNR 下降超过 `0.05 dB`；
2. SSIM 下降超过 `0.001`，或 LPIPS 上升超过 `0.001`。

达到判据后，在当前评估检查点停止。当前检查点仍被保存，但默认推荐下降前的检查点。全局最佳检查点按 PSNR 最高优先选择；PSNR 相同时依次选择 SSIM 更高、LPIPS 更低的检查点。

## 实时进度与输出

每次运行使用唯一 `run_id`，输出位于：

`/home/pch/myGS/outputs/visible/<run_id>/`

运行目录至少包含：

- `status.json`：当前阶段、迭代、进度、损失、时间、PID 和最后更新时间。
- `metrics.csv`：各评估点的 PSNR、SSIM、LPIPS、相对变化和判定结果。
- `summary.json`：结束状态、停止原因、最后迭代、最佳迭代和最佳指标。
- `point_cloud/iteration_<N>/point_cloud.ply`：官方点云输出。
- `chkpnt<N>.pth`：可恢复训练的 checkpoint。
- `evaluation/iteration_<N>/`：固定测试视角的渲染图和对应真值图。
- `train.pid`：持久训练进程 PID。

完整控制台日志位于：

`/home/pch/myGS/logs/<timestamp>-train-visible.log`

训练以服务器持久进程运行。SSH 会话中断不会终止训练；后续通过 `status.json`、`metrics.csv`、日志和 PID 检查运行状态。状态文件采用临时文件加原子替换方式刷新，避免读取到半写入 JSON。

## 异常处理与恢复

- 输入、环境、源码提交或 LPIPS 权重预检失败：正式训练不启动，`status.json` 标记 `preflight_failed`。
- CUDA OOM 或训练进程异常：记录退出码、最后迭代和最后有效 checkpoint，状态标记 `failed`。
- 评估异常：不使用缺失指标继续早停判断；保存错误并停止本次运行，避免把不完整结果视为正常结果。
- SSH 中断：训练进程继续运行，监控脚本可重新附着到状态和日志。
- 人工停止：仅向明确记录的训练 PID 发送信号，不使用进程名批量终止。
- 恢复训练：只从同一运行目录的最后有效 checkpoint 恢复，并保留既有指标历史。
- 已生成结果不自动删除；需要删除时遵守一次只删除一个明确文件的项目约束。

## 验证标准

正式运行前必须通过：

- Shell 语法检查。
- Python 自动化测试，覆盖阈值边界、下降组合、最佳迭代选择和状态序列化。
- 补丁基线提交检查和幂等检查。
- LPIPS VGG 权重加载测试。
- 小迭代 smoke run，使用仅限验证的缩短迭代与评估间隔，验证状态、日志、checkpoint 和一次评估路径可以完整产生；正式运行参数仍固定为 5000 步开始、每 1000 步评估、最多 15000 步。

正式运行完成或早停后必须验证：

- `status.json` 为 `completed` 或 `early_stopped`，而不是运行中或失败。
- 从 5000 步开始的每个已到达检查点均有完整指标记录。
- `summary.json` 中停止原因、最后迭代和最佳迭代一致。
- 最佳 checkpoint、point cloud 和固定视角渲染图存在。
- thermal 没有启动新的训练进程或生成本次运行输出。
- 原始数据未被移动、删除、改名或修改。
